# Document 03: Gold Mask Format & Package Specification

---

## 1. The Gold Standard (Authoritative Format)

| Property | Requirement |
|----------|-------------|
| File type | PNG only (never JPG, never WebP) |
| Channels | 1-channel grayscale (mode `L`). No alpha. No palette for binary masks |
| Values | Hard binary: exactly {0, 255}. Any other value = automatic rejection (QC-002) |
| Dimensions | Exactly equal to source width × height. No resize/crop/pad (QC-001) |
| Anti-aliasing | None. Nearest-neighbor everywhere in the pipeline |
| Compression | PNG lossless (zlib level 6); never lossy |
| Semantics | White = the named body part's visible pixels; black = everything else |
| Bit depth | 8-bit for binary masks; 16-bit for `label_map_part.png` |

Model-input-resized masks may exist only under `work\` and are never part of the gold package.

## 2. Gold Package Layout (per image)

```
data\packages\<image_id>\
  source.png                      # canonical copy (or source.jpg if original was JPG — masks still PNG)
  label_map_part.png              # 16-bit indexed master PART map (doc 02 §2)
  label_map_material.png          # 8-bit indexed MATERIAL map (doc 02 §3)
  masks\                          # generated binary atomics: left_forearm.png, right_toes.png, ...
  masks_regions\                  # band regions: waist.png, body_contact_region.png, ...
  masks_derived\                  # unions: left_hand.png, visible_body_skin.png, ...
  projected\                      # left_breast_projected_region.png, amodal_left_forearm.png, ...
  inpaint\                        # inpaint_left_hand_d8f4.png, ... (derived edit masks, §6)
  protected\                      # background.png, face_protected.png, clothing_visible.png (QA copies)
  matting\                        # hair_binary.png (copy), hair_trimap.png, hair_alpha_matte.png (§7)
  overlays\                       # per-mask color overlays on source (evidence only)
  qa_panels\                      # boundary zoom panels per hard mask (doc 09 §6)
  crops\                          # hand/foot/chest/face crops + crop_to_full_transform.json (§5)
  annotations\
    image_annotations.json        # CVAT/COCO export (polygons/RLE) — regenerable evidence
    cvat_task_backup.zip          # raw CVAT task export
  manifest.json                   # THE authority (doc 04 §1)
  qa_report.json                  # auto-QA + VLM verdicts (doc 04 §2)
```

**AMENDED (doc 17 §6):** for a multi-person image, this entire layout above is nested one level
deeper, once per promoted person: `data\packages\<image_id>\instances\p0\...`,
`...\instances\p1\...`, plus one new small top-level `image_manifest.json` indexing all
instances. A single-person image is the trivial N=1 case — exactly one `instances\p0\` folder.
Nothing above this line changes; see doc 17 §6 for the full layout and `image_manifest.json`
schema.

## 3. Naming Rules

- `<image_id>` = `img_` + first 12 hex of source SHA-256 (stable, collision-safe, no PII).
- Mask filenames = exact ontology label + `.png`, lowercase snake_case, no spaces.
- Never encode status in filenames; status lives in manifest. Never overwrite gold — corrections
  create a new version folder `masks@v2\` promoted atomically by the packager, previous kept until
  `maskfactory gc` (runbook §6).

## 4. Master Maps → Binary Masks (Exclusive-by-Construction)

1. Human/pipeline edits happen as polygons/brush in CVAT or as map edits.
2. `maskfactory fuse` resolves everything into `label_map_part.png` + `label_map_material.png`
   using z-order (doc 07 S09). These two files are the only place exclusivity is decided.
3. `maskfactory export-binaries` regenerates ALL binary PNGs from the maps. Binary files are
   therefore views; hand-editing a binary PNG directly is prohibited (checksum mismatch vs map
   triggers QC-030).
4. Atomic↔atomic overlap is impossible by construction; QC-011 still verifies as belt-and-suspenders.

## 5. Multi-Scale Crop Contract

Hard regions are annotated on crops, not the full image:
- `crops\left_hand_crop.png` (etc. for right_hand, feet, chest, face/hairline) — square crop,
  side = 1.6 × part bbox max side, resized to 1024 with **nearest/Lanczos pairing rule**: image
  Lanczos, masks nearest.
- `crops\crop_to_full_transform.json`: `{part, x0, y0, scale, crop_size, source_sha256}` —
  exact affine placement back to full res. Reprojection uses nearest-neighbor; QC-018 verifies
  round-trip IoU ≥ 0.995 between crop-space mask and full-res mask within the crop window.

## 6. Gold vs Inpaint Separation (Never Mix)

- Gold = exact visible pixels. Never dilated/feathered/blurred.
- Inpaint derivatives generated by `maskfactory derive-inpaint`:
  `inpaint_<label>_d<dilate_px>f<feather_px>.png` — grayscale allowed here (feather ramp 0–255),
  because inpaint masks are NOT truth. Defaults `d8f4` @1024 ref (scaled by image size);
  per-label overrides in `configs\inpaint.yaml`. Settings + source gold hash recorded in manifest
  `inpaint_derivatives[]`. ComfyUI consumes these (doc 13).

## 7. Hair / Soft-Boundary Exception

Binary `hair.png` remains the gold authority. Additionally stored when hair is prominent
(auto-trigger: hair area ≥ 2% of person bbox):
- `hair_trimap.png` — {0 bg, 128 unknown band (dilate binary ±6 px scaled), 255 fg}
- `hair_alpha_matte.png` — 0–255 matte from ViTMatte/BiRefNet-matting seeded by the trimap
Same optional treatment for `lace_or_sheer` clothing regions. Mattes are evidence/compositing
aids; all metrics and training labels use the binary unless a matting-specific model is trained (v2).

## 8. Overlay & Evidence Files

- Overlay = source at 100% + mask fill RGBA(255,64,64,110) + 1 px hard contour; per-label color
  table in `configs\viz.yaml` for multi-part overlays. PNG, same dims (allowed RGBA — evidence only).
- Annotation exports (COCO RLE + CVAT XML) kept so masks can be regenerated; they are evidence,
  never authority — on conflict, `label_map_part.png` wins.

## 9. Integrity & Reproducibility

- Every file in the package is SHA-256 hashed into `manifest.files{}` at export; packager verifies
  before status can become `human_approved_gold`.
- Package is immutable after approval; any edit spawns v2 and demotes v1 to `deprecated`.
- `maskfactory verify-package <image_id>` re-checks all hashes + reruns format QCs; used by backup
  restore tests (runbook §5) and nightly integrity sweep.
